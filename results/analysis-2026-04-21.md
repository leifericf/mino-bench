# Performance Analysis: 2026-04-21

## Test environment
- macOS Darwin 24.6.0, Apple Silicon
- mino commit: 14455e6 (perf-benchmark-optimize)
- cc -std=c99 -O2

## C API vs mino eval overhead

| Operation | C API (us) | Mino eval (us) | Ratio |
|-----------|-----------|-----------------|-------|
| (+ 1 2) | 1.25 | 6.92 | 5.5x |
| integer creation | 0.10 | 6.04 | 60x |
| symbol lookup (interned) | 0.01 | 6.22 | 622x |
| cons creation | 0.19 | 6.53 | 34x |
| get {:a 1 :b 2 :c 3} :b | 6.70 | 6.46 | ~1x |
| loop/recur (per iter) | 1.91 | 2.95 | 1.5x |
| rangev 100 | 17.7 | 28.4 | 1.6x |
| mapv inc (rangev 100) | 754 | 876 | 1.2x |
| filterv odd? (rangev 100) | 1410 | 1670 | 1.2x |
| fib(20) | 58700 | 52340 | ~1x |

Key finding: ~6us floor on every mino operation regardless of complexity.

## Top 10 first-order issues

### 1. Environment lookup is O(n) with strcmp per binding
- env_find_here (runtime_env.c:76): linear scan of all bindings with strcmp
- Root env has ~200+ bindings (all core primitives)
- Every variable reference walks this list
- Evidence: C mino_symbol = 0.01us, mino (symbol "+") = 6.22us (622x gap)

### 2. eval_symbol copies name to stack buffer + strcmp chain
- eval_special.c:28-87: memcpy to 256-byte stack buffer, then dyn_lookup + env_get
- For simple "+": memcpy + dyn_lookup miss + env_find_here full scan
- Evidence: global var lookup = 6.01us despite being a single pointer dereference in concept

### 3. Per-iteration allocation pressure in loop/recur
- Each loop/recur iteration allocates a new environment frame
- loop 100 iters = 301us = 3.0us/iter
- dotimes 100 = 707us = 7.1us/iter (macro expanding to let+loop+inc)

### 4. Lazy sequence overhead (range/map/filter)
- doall (range 100) = 1.08ms vs rangev 100 = 28us (39x slower)
- Each lazy step: thunk creation + fn call + env alloc
- map inc (range 100) lazy = 1.94ms vs mapv inc (rangev 100) = 876us

### 5. alias_resolve is O(n) linear scan
- eval_special.c:18-26: scans alias table with strcmp for every qualified symbol
- Part of the 6us floor for every namespace-qualified name

### 6. dyn_lookup scans all dynamic bindings on every symbol eval
- Called before env_get for every symbol, even non-dynamic ones
- Most symbols are not dynamically bound

### 7. Small collection construction is allocation-dominated
- [1 2 3] = 6.70us, {:a 1} = 7.38us
- Eval-per-element adds overhead vs single allocation

### 8. String operations allocate excessively
- str concat (3 short) = 9.52us, format "%d" 42 = 9.69us
- Intermediate string allocations per concat step

### 9. map_get_val doesn't exploit keyword interning for pointer equality
- Map get on interned keys could use pointer identity fast-path

### 10. Function call frame allocation
- Each fn call allocates new env frame for parameters
- 0-arg = 6.53us, 3-arg = 8.59us, 5-arg = 9.80us (~1.3us/arg)
